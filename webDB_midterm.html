인터넷 vs 웹 ★
Ch.1 Section 01 · 02
웹 = 인터넷이 아니다. 웹은 인터넷 위에서 돌아가는 서비스 중 하나일 뿐!
인터넷
  • 하드웨어적 개념 — 컴퓨터들을 연결한 네트워크
  • 미국 국방부의 ARPANET이 시초
  • 이메일, FTP, 텔넷, 웹 등 여러 서비스가 인터넷 위에서 동작
웹 (WWW)
  • 인터넷 서비스 중 하나
  • HTTP 프로토콜 사용
  • HTML 문서를 연결해서 정보 제공
  • 개발 목적: 연구자들이 흩어진 정보를 쉽게 공유

인터넷 기반 서비스 종류

이메일, FTP, Telnet, DNS, 웹 등 — 다 TCP/IP를 사용하지만 응용 계층에서 각자 별도 프로토콜을 가짐.

HTTP · URL
Ch.1 Section 01

HTTP (HyperText Transfer Protocol)

클라이언트(브라우저)와 서버가 HTML 문서를 주고받기 위한 통신 규약.

URL 구조

http :// www.naver.com : 80 / news/index.html ① ② ③ ④ ① 프로토콜 — http 또는 https ② 서버 주소 — 서버가 위치한 주소 ③ 포트번호 — 웹 서비스 기본값 80 (생략 가능) ④ 파일 경로 — 없으면 index.html이 기본

포트 (Port)

네트워크 서비스를 제공하기 위한 일종의 출입문. 하나의 컴퓨터에서 여러 서비스를 구분하는 데 사용. 포트와 프로토콜이 일치하지 않으면 서비스 이용 불가.

도메인 네임 시스템 (DNS)

IP 주소는 숫자 (예: 123.123.123.123). 도메인 이름(www.naver.com)을 IP 주소로 변환하는 시스템이 DNS.

클라이언트-서버 구조 ★
Ch.1 Section 03
클라이언트
  • 웹 서비스를 이용하는 사용자 쪽
  • 브라우저가 동작
  • 프런트엔드: HTML, CSS, JavaScript
서버
  • 웹 서비스를 제공하는 쪽
  • 웹 서버 소프트웨어 동작
  • 백엔드: 서블릿/JSP, DB
브라우저 → HTTP 요청 → 웹서버(톰캣) → 서블릿 실행 → DB 브라우저 ← HTML 응답 ← 서블릿이 HTML 생성
"프런트엔드, 백엔드 모르는 사람 없죠? 없으면 넘어갑니다."
정적 vs 동적 웹 ★
Ch.1 Section 03
정적(Static) 웹
  • HTML 파일을 그대로 전송
  • 서버는 찾아서 보낼 뿐, 처리 없음
  • 누가 접속해도 항상 같은 페이지
동적(Dynamic) 웹
  • 서버에서 Java 코드가 실행됨
  • 사용자 입력에 따라 HTML을 그때그때 생성
  • 로그인, 쇼핑몰, 검색 결과 등
클라이언트 동적(JS로 화면 바꾸기)과 서버 동적(서블릿이 HTML 생성)은 다른 개념!

Server Script vs Client Script

Client Script

브라우저에서 실행. 대표: JavaScript

Server Script

서버에서 실행. 대표: JSP, ASP, PHP

교수님 설명: "환영합니다가 박혀있는 페이지 → 정적. 내 이름 입력하면 '환영합니다, 김OO'처럼 서버가 HTML을 그때 만들어서 보내주는 것 → 동적"
네트워크 · 프로토콜
Ch.1 Section 02

TCP/IP

  • 컴퓨터 간 통신 프로토콜 중 가장 널리 쓰임
  • 하드웨어, OS, 접속 매체 관계없이 동작하는 개방형 구조
  • TCP: 데이터 흐름 관리, 정확성 확인
  • IP: 데이터(패킷)를 목적지까지 전송

OSI 7계층 → TCP/IP 4계층으로 단순화

IP 주소

TCP/IP 네트워크에서 컴퓨터를 구분하는 주소. 123.123.123.123 형태 (4개, 10진수).

HTML · CSS · JS 역할
Ch.1 Section 03
HTML

웹 문서의 구조를 정의. 태그로 콘텐츠에 의미 부여.

CSS

웹 문서의 표현(디자인)을 담당. 위치, 색상, 크기 등.

JavaScript

HTML 문서에서 동적으로 변하는 콘텐츠를 처리. 이벤트 처리, 서버와 통신.

HTML5 원칙: HTML = 구조(뼈대), CSS = 표현(디자인). 두 역할을 분리하자!
왜 Java인가? ★
Ch.2 Section 01
"수많은 언어 중에 왜 Java를 쓰냐? 이유를 알고 써야지 그냥 쓰면 안 된다."
⭐ 핵심 이유: 이식성(Portability) — JVM 덕분에 어떤 OS에서도 코드 변경 없이 동작

인터넷 환경은 Windows, Linux, Mac, Android, iOS 등 다양한 OS가 공존. C 언어로 만든 프로그램은 OS가 바뀌면 다시 컴파일해야 하지만, Java는 JVM만 있으면 어디서든 동일하게 동작.

"리눅스로 서버 운영하다가 유닉스로 바꿔도 Java 코드는 그대로 쓸 수 있다. C였으면 다시 만들어야 한다."

Java의 특징

  • 간결하면서도 강력한 객체지향 프로그래밍 언어
  • 플랫폼에 독립적 (이식성)
  • 많은 오픈소스 라이브러리 → 생산성 향상
  • GUI 기반 응용 프로그램 개발에는 적합하지 않음
  • 하드웨어 정밀 제어 프로그램에는 비교적 적합하지 않음

Java 플랫폼

API + JVM으로 구성. Java SE(일반), Java EE(대규모 개발) 등이 있음. 웹 개발은 톰캣 같은 서블릿 컨테이너만 설치해도 충분.

JVM · 바이트코드 ★
Ch.2 Section 01
C 언어 실행 과정
소스(.c) ↓ 컴파일 (컴파일러) 오브젝트(.obj) ↓ 링크 실행파일(.exe) ↓ OS가 직접 실행 → OS가 다르면 다시!
Java 실행 과정
소스(.java) ↓ javac (컴파일) 바이트코드(.class) ↓ JVM이 실행 (OS 위에 JVM 설치) → JVM만 있으면 OK!
바이트코드 = 기계어가 아님. JVM이 이해하는 언어. JVM이 각 OS에 맞게 번역해서 실행.

JVM이 중요한 이유

OS 위에 JVM이 올라가 있어서 OS의 영향을 받지 않음. Write Once, Run Anywhere.

"Java 소스는 컴파일 후 바이트코드가 생성되며, JVM은 바이트코드를 해석하여 운영체제에서 실행할 수 있도록 번역한다."

DOM · 이벤트 처리 ★
Ch.3 Section 03

DOM (Document Object Model)

HTML 문서를 프로그램적으로 처리할 수 있도록 문서 구조 전체를 객체화한 것. JS가 이를 통해 HTML 요소에 접근 가능.

DOM 접근

document.getElementById("myId") // id로 찾기 document.querySelector(".myClass") // CSS 셀렉터로 찾기

이벤트 처리 3가지 방법

// ① HTML 태그에 속성 <button onclick="fn()">클릭</button> // ② DOM 속성 document.getElementById("btn").onclick = fn; // ③ addEventListener (권장) element.addEventListener("click", fn);
서블릿이란? ★
Ch.4 Section 01
⭐ 서블릿 = Server + Applet = 서버에서 실행되는 작은 Java 프로그램

클라이언트 요청을 받아서 처리하고 HTML 응답을 돌려주는 역할. Java로 만들기 때문에 Java의 특징(이식성, API 사용)을 그대로 가짐.

서블릿의 동작

브라우저 → HTTP 요청 → 톰캣(서블릿 컨테이너) → 해당 서블릿 실행 → DB 액세스 (필요 시) → HTML 동적 생성 → 브라우저에 응답

서블릿 장단점

장점
  • Java API 모두 사용 가능
  • OS/하드웨어 독립적
  • 스레드 기반 다중 처리

서블릿 생명주기 ★
Ch.4 Section 01
⭐ 핵심: init()은 최초 1회. service()는 매 요청마다.
첫 번째 요청: 컴파일 (.java → .class) → init() // 최초 1회, 초기화 → service() // 매 요청마다 실행 → doGet() or doPost() 두 번째 이후 요청: service() → doGet()/doPost() // init() 생략! 코드 수정 / 서버 종료: destroy() // 최종 1회, 정리
메서드호출 시점용도
init()최초 1회초기화 작업 (DB 연결 등)
service()매 요청마다GET/POST 분기
doGet()GET 요청 시실제 처리 로직
doPost()POST 요청 시실제 처리 로직
destroy()종료 시 1회정리 작업
service()를 오버라이딩하면 → doGet/doPost 자동 실행 안 됨
service() 안 건드리면 → doGet/doPost 자동으로 실행됨
"처음 요청하면 약간 delay가 있다. 컴파일, init(), 컨테이너에 올리는 시간 때문. 두 번째부터는 바로바로 나온다."
서블릿 클래스 구조 ★
Ch.4 Section 01
LoginServlet.java
import javax.servlet.*; import javax.servlet.http.*; import java.io.*; @WebServlet("/LoginServlet") // URL 매핑: 이 URL 요청 → 이 서블릿 실행 // form의 action="LoginServlet"과 반드시 일치! public class LoginServlet extends HttpServlet { // HttpServlet 상속 = 서블릿이 되는 조건 protected void doGet( HttpServletRequest request, // 요청 정보 객체 HttpServletResponse response) // 응답 출력 객체 throws ServletException, IOException { // 왜 request, response 두 개를 받는가? // 요청 처리 + 응답 = 두 역할이 필요하기 때문! // 1. 먼저! 한글 깨짐 방지 response.setContentType("text/html; charset=utf-8"); // 2. 파라미터 추출 String id = request.getParameter("id"); // 3. 브라우저로 HTML 출력 PrintWriter out = response.getWriter(); out.print("<html><body>"); out.print("<h2>안녕, " + id + "!</h2>"); out.print("</body></html>"); } }
response.setContentType()은 반드시 getWriter() 호출 전에!

GET vs POST ★
Ch.4 — 교수님이 "언제 뭘 써야 하는지 모르면 기술이 없는 것"이라고 강조!
"GET 쓴 사람, POST 쓴 사람 모두 왜 그걸 선택했는지 설명 못하면 기술 없는 것."
GET 방식
  • 데이터가 URL(쿼리스트링)에 포함
  • 예) ...?id=kim&pw=1234
  • 장점: 빠름
  • 단점: URL에 노출됨, 길이 제한
  • 서블릿: doGet() 실행
POST 방식
  • 데이터가 HTTP Body에 포함
  • URL에 안 보임
  • 장점: 보안 우수, 길이 제한 없음
  • 단점: GET보다 느림
  • 서블릿: doPost() 실행
로그인/회원가입처럼 민감한 정보 → POST. 검색어처럼 노출돼도 되는 것 → GET.
폼 → 서블릿 전체 흐름 ★
Ch.4 실습
login.html
<form action="LoginServlet"> 아이디: <input type="text" name="id"> 비밀번호: <input type="password" name="pw"> <input type="submit" value="로그인"> </form> <!-- 전송 시 생성되는 URL: --> <!-- .../LoginServlet?id=kim&pw=1234 -->
LoginServlet.java — doGet()
response.setContentType("text/html; charset=utf-8"); String id = request.getParameter("id"); // "kim" String pw = request.getParameter("pw"); // "1234" PrintWriter out = response.getWriter(); if ("admin".equals(id) && "1234".equals(pw)) { out.print("<h2>로그인 성공!</h2>"); } else { out.print("<h2>로그인 실패</h2>"); }
로그인 폼(login.html)은 서블릿이 아닌 HTML 파일로 만들어야 한다! 정적 파일을 서블릿으로 만들면 컴파일/init/service 불필요하게 실행 → 서버 낭비.

왜 정보 공유가 필요한가 ★
Ch.5 Section 01
HTTP는 비연결형(Stateless) 통신. 페이지를 이동하면 이전 페이지의 정보가 사라짐!

로그인 상태 유지, 장바구니, 쇼핑몰의 주문 정보 등 — 여러 페이지에 걸쳐 정보를 공유해야 하는 경우가 많음.

정보 공유 5가지 방법

방법저장 위치언제 쓰나
URL RewritingURL간단한 페이지 이동, 노출 괜찮은 데이터
Cookie클라이언트(브라우저)방문 횟수, 비민감 정보 유지
Session서버로그인 상태, 민감한 정보
ServletContext서버 (공유 공간)모든 사용자 공통 데이터
RequestDispatcherrequest 객체서블릿 간 데이터 전달 (forward)
"언제 어떤 기술을 써야 하는지 판단할 수 있어야 한다. 그게 설계 능력이다."
sendRedirect ★
Ch.5 Section 01

서블릿에서 다른 페이지로 이동시키는 메서드. HTTP Response 객체에 있음.

response.sendRedirect("main.jsp"); response.sendRedirect("http://www.naver.com"); // 주의: sendRedirect 후에도 아래 코드가 실행됨! // 반드시 return; 함께 사용 response.sendRedirect("login.html"); return;

동작 방식

서버 → 브라우저에게 "이 URL로 가라" (302 응답) → 브라우저가 해당 URL로 새 요청 → URL 주소 바뀜. 이전 request 데이터 소멸.
URL Rewriting ★
Ch.5 Section 01

URL에 쿼리스트링을 붙여서 다음 페이지로 데이터 전달.

쿼리스트링 (Query String)

URL에 부가해서 데이터 전달 목적으로 사용하는 문자열.

URL?name=value&name2=value2 ↑ ↑ 물음표 각 데이터는 &로 연결
예시
// sendRedirect에 쿼리스트링 붙이기 String name = request.getParameter("name"); String gender = "남자"; response.sendRedirect("ResultServlet?name=" + name + "&gender=" + gender); // 받는 쪽 서블릿에서: String name = request.getParameter("name"); String gender = request.getParameter("gender");
한글은 URL 인코딩 필요! URLEncoder.encode(value, "UTF-8") 사용.

장점: 간단. 단점: URL에 노출됨. 매 페이지마다 붙여야 해서 복잡.

쿠키 (Cookie) ★
Ch.5 Section 02
⭐ 쿠키 = 서버가 클라이언트(브라우저)에 저장하는 데이터. name=value 형식.

쿠키 생성 & 전송

Cookie ck = new Cookie("name", "kim"); // 생성자: 첫 번째=키 이름, 두 번째=값 (둘 다 String) ck.setMaxAge(3600); // 유효시간(초). 0이면 즉시 삭제. 음수=브라우저 닫을 때 삭제 response.addCookie(ck); // 브라우저에 저장 명령

쿠키 읽기

Cookie[] cookies = request.getCookies(); // 모든 쿠키 배열로 if (cookies != null) { for (Cookie c : cookies) { if (c.getName().equals("name")) { String val = c.getValue(); } } }

쿠키 삭제

Cookie ck = new Cookie("name", ""); ck.setMaxAge(0); // 0 = 즉시 삭제 response.addCookie(ck);

쿠키 값 변경

ck.setValue("newValue"); response.addCookie(ck);
장점
  • 서버 메모리 사용 안 함
  • 브라우저 닫아도 유지 가능 (setMaxAge 설정 시)
단점
  • 클라이언트 저장 → 보안 취약
  • 개발자도구에서 바로 보임/수정 가능
  • 4KB 용량 제한
"쿠키는 서버가 저장 요청을 하면 클라이언트 브라우저가 관리하는 특정 공간에 보관. 개발자 도구에서 다 보이고 삭제, 변경도 가능. 그래서 보안에 취약."
세션 (Session) ★
Ch.5 Section 03
⭐ 세션 = 서버에 저장되는 데이터. 브라우저마다 세션 ID가 발급됨.

세션 생성 & 저장

HttpSession session = request.getSession(); // getSession(true) : 없으면 새로 생성 (기본값) // getSession(false): 없으면 null 반환 // 저장 (두 번째 인자가 Object 타입!) session.setAttribute("loginUser", "kim");

세션 읽기

// getAttribute()의 반환 타입 = Object → 형변환 필수! String user = (String) session.getAttribute("loginUser"); // 또는 toString() 사용 String user = session.getAttribute("loginUser").toString();

세션 삭제

session.removeAttribute("loginUser"); // 단일 삭제 session.invalidate(); // 전체 삭제 (로그아웃)

유효시간 설정

session.setMaxInactiveInterval(1800); // 초 단위. 기본 1800초(30분) // 0 또는 음수 = 유효시간 없음 (주의: 메모리 누수)
장점
  • 서버에 저장 → 보안 우수
  • 용량 제한 없음
단점
  • 서버 메모리 사용
  • 동시 접속자 많으면 서버 부담
  • 브라우저 단위 — 브라우저 다르면 별도 세션
세션은 브라우저 단위! 크롬에서 로그인해도 엣지로 열면 로그아웃 상태. 방문 횟수처럼 지속돼야 하는 건 쿠키로!
"세션이라는 것은 클라이언트가 접속하면 서버 쪽에 생성되는 공간. 브라우저마다 세션 ID가 발급되고 그걸 기준으로 클라이언트 관리."
ServletContext
Ch.5 Section 04

서블릿 컨테이너에서 모든 서블릿이 공유하는 메모리 공간. 웹 애플리케이션 단위로 만들어짐. Java의 클래스 변수(static)와 유사.

ServletContext ctx = request.getServletContext(); ctx.setAttribute("notice", "서버 점검 예정"); String notice = (String) ctx.getAttribute("notice");
모든 사용자가 공통으로 봐야 하는 데이터(공지사항 등)에 적합. Object 타입이라 형변환 필요.
RequestDispatcher
Ch.5 Section 05

request, response 객체를 그대로 다른 서블릿에 전달. sendRedirect와 달리 서버 내부에서 이동.

RequestDispatcher rd = request.getRequestDispatcher("/ResultServlet"); rd.forward(request, response); // request 객체가 그대로 넘어감! // 다음 서블릿에서도 getParameter() 사용 가능
sendRedirect
  • 브라우저가 새 요청
  • URL 주소 바뀜
  • request 데이터 소멸
  • 외부 URL로도 이동 가능
forward (dispatcher)
  • 서버 내부에서 전달
  • URL 주소 안 바뀜
  • request 데이터 유지!
  • 같은 서버 내에서만
5가지 방법 비교 ★
Ch.5 총정리 — 시험 유력 출제 범위!
방법저장 위치보안용량유지 기간적합한 용도
URL Rewriting URL 취약 제한적 한 번의 이동 간단한 값 전달
Cookie 클라이언트 취약 4KB setMaxAge 설정값 방문 횟수, 자동 로그인

RequestDispatcher request 객체 우수 제한 없음 요청 처리 동안 서블릿 간 처리 위임
⭐ 핵심 차이: 쿠키=클라이언트 저장 / 세션=서버 저장 / 세션은 브라우저 단위!
"각 기술의 장단점을 이해해야지만, 이런 상황에서는 이 기술을 써야지 라는 판단을 할 수 있다. 그게 설계 능력이다."
