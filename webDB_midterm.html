인터넷 vs 웹 ★
Ch.1 Section 01 · 02
웹 = 인터넷이 아니다. 웹은 인터넷 위에서 돌아가는 서비스 중 하나일 뿐!
인터넷
  • 하드웨어적 개념 — 컴퓨터들을 연결한 네트워크
  • 미국 국방부의 ARPANET이 시초
  • 이메일, FTP, 텔넷, 웹 등 여러 서비스가 인터넷 위에서 동작
웹 (WWW)
  • 인터넷 서비스 중 하나
  • HTTP 프로토콜 사용
  • HTML 문서를 연결해서 정보 제공
  • 개발 목적: 연구자들이 흩어진 정보를 쉽게 공유

인터넷 기반 서비스 종류

이메일, FTP, Telnet, DNS, 웹 등 — 다 TCP/IP를 사용하지만 응용 계층에서 각자 별도 프로토콜을 가짐.

HTTP · URL
Ch.1 Section 01

HTTP (HyperText Transfer Protocol)

클라이언트(브라우저)와 서버가 HTML 문서를 주고받기 위한 통신 규약.

URL 구조

http :// www.naver.com : 80 / news/index.html ① ② ③ ④ ① 프로토콜 — http 또는 https ② 서버 주소 — 서버가 위치한 주소 ③ 포트번호 — 웹 서비스 기본값 80 (생략 가능) ④ 파일 경로 — 없으면 index.html이 기본

포트 (Port)

네트워크 서비스를 제공하기 위한 일종의 출입문. 하나의 컴퓨터에서 여러 서비스를 구분하는 데 사용. 포트와 프로토콜이 일치하지 않으면 서비스 이용 불가.

도메인 네임 시스템 (DNS)

IP 주소는 숫자 (예: 123.123.123.123). 도메인 이름(www.naver.com)을 IP 주소로 변환하는 시스템이 DNS.

클라이언트-서버 구조 ★
Ch.1 Section 03
클라이언트
  • 웹 서비스를 이용하는 사용자 쪽
  • 브라우저가 동작
  • 프런트엔드: HTML, CSS, JavaScript
서버
  • 웹 서비스를 제공하는 쪽
  • 웹 서버 소프트웨어 동작
  • 백엔드: 서블릿/JSP, DB
브라우저 → HTTP 요청 → 웹서버(톰캣) → 서블릿 실행 → DB 브라우저 ← HTML 응답 ← 서블릿이 HTML 생성
"프런트엔드, 백엔드 모르는 사람 없죠? 없으면 넘어갑니다."
정적 vs 동적 웹 ★
Ch.1 Section 03
정적(Static) 웹
  • HTML 파일을 그대로 전송
  • 서버는 찾아서 보낼 뿐, 처리 없음
  • 누가 접속해도 항상 같은 페이지
동적(Dynamic) 웹
  • 서버에서 Java 코드가 실행됨
  • 사용자 입력에 따라 HTML을 그때그때 생성
  • 로그인, 쇼핑몰, 검색 결과 등
클라이언트 동적(JS로 화면 바꾸기)과 서버 동적(서블릿이 HTML 생성)은 다른 개념!

Server Script vs Client Script

Client Script

브라우저에서 실행. 대표: JavaScript

Server Script

서버에서 실행. 대표: JSP, ASP, PHP

교수님 설명: "환영합니다가 박혀있는 페이지 → 정적. 내 이름 입력하면 '환영합니다, 김OO'처럼 서버가 HTML을 그때 만들어서 보내주는 것 → 동적"
네트워크 · 프로토콜
Ch.1 Section 02

TCP/IP

  • 컴퓨터 간 통신 프로토콜 중 가장 널리 쓰임
  • 하드웨어, OS, 접속 매체 관계없이 동작하는 개방형 구조
  • TCP: 데이터 흐름 관리, 정확성 확인
  • IP: 데이터(패킷)를 목적지까지 전송

OSI 7계층 → TCP/IP 4계층으로 단순화

IP 주소

TCP/IP 네트워크에서 컴퓨터를 구분하는 주소. 123.123.123.123 형태 (4개, 10진수).

HTML · CSS · JS 역할
Ch.1 Section 03
HTML

웹 문서의 구조를 정의. 태그로 콘텐츠에 의미 부여.

CSS

웹 문서의 표현(디자인)을 담당. 위치, 색상, 크기 등.

JavaScript

HTML 문서에서 동적으로 변하는 콘텐츠를 처리. 이벤트 처리, 서버와 통신.

HTML5 원칙: HTML = 구조(뼈대), CSS = 표현(디자인). 두 역할을 분리하자!
왜 Java인가? ★
Ch.2 Section 01
"수많은 언어 중에 왜 Java를 쓰냐? 이유를 알고 써야지 그냥 쓰면 안 된다."
⭐ 핵심 이유: 이식성(Portability) — JVM 덕분에 어떤 OS에서도 코드 변경 없이 동작

인터넷 환경은 Windows, Linux, Mac, Android, iOS 등 다양한 OS가 공존. C 언어로 만든 프로그램은 OS가 바뀌면 다시 컴파일해야 하지만, Java는 JVM만 있으면 어디서든 동일하게 동작.

"리눅스로 서버 운영하다가 유닉스로 바꿔도 Java 코드는 그대로 쓸 수 있다. C였으면 다시 만들어야 한다."

Java의 특징

  • 간결하면서도 강력한 객체지향 프로그래밍 언어
  • 플랫폼에 독립적 (이식성)
  • 많은 오픈소스 라이브러리 → 생산성 향상
  • GUI 기반 응용 프로그램 개발에는 적합하지 않음
  • 하드웨어 정밀 제어 프로그램에는 비교적 적합하지 않음

Java 플랫폼

API + JVM으로 구성. Java SE(일반), Java EE(대규모 개발) 등이 있음. 웹 개발은 톰캣 같은 서블릿 컨테이너만 설치해도 충분.

JVM · 바이트코드 ★
Ch.2 Section 01
C 언어 실행 과정
소스(.c) ↓ 컴파일 (컴파일러) 오브젝트(.obj) ↓ 링크 실행파일(.exe) ↓ OS가 직접 실행 → OS가 다르면 다시!
Java 실행 과정
소스(.java) ↓ javac (컴파일) 바이트코드(.class) ↓ JVM이 실행 (OS 위에 JVM 설치) → JVM만 있으면 OK!
바이트코드 = 기계어가 아님. JVM이 이해하는 언어. JVM이 각 OS에 맞게 번역해서 실행.

JVM이 중요한 이유

OS 위에 JVM이 올라가 있어서 OS의 영향을 받지 않음. Write Once, Run Anywhere.

"Java 소스는 컴파일 후 바이트코드가 생성되며, JVM은 바이트코드를 해석하여 운영체제에서 실행할 수 있도록 번역한다."

DOM · 이벤트 처리 ★

DOM (Document Object Model)

HTML 문서를 프로그램적으로 처리할 수 있도록 문서 구조 전체를 객체화한 것. JS가 이를 통해 HTML 요소에 접근 가능.

document.getElementById("myId") // id로 찾기 document.querySelector(".myClass") // CSS 셀렉터로 찾기

// ① HTML 태그에 속성 <button onclick="fn()">클릭</button> // ② DOM 속성 document.getElementById("btn").onclick = fn; // ③ addEventListener (권장) element.addEventListener("click", fn);
서블릿이란? ★
Ch.4 Section 01
⭐ 서블릿 = Server + Applet = 서버에서 실행되는 작은 Java 프로그램

클라이언트 요청을 받아서 처리하고 HTML 응답을 돌려주는 역할. Java로 만들기 때문에 Java의 특징(이식성, API 사용)을 그대로 가짐.

교수님 강조: 서블릿은 멀티 스레드(Multi-Thread) 방식으로 동작합니다. 요청마다 프로세스를 새로 만드는 것이 아니라, 메모리에 로드된 객체를 여러 스레드가 공유하므로 서버 부하가 적고 빠릅니다.

서블릿의 동작

브라우저 → HTTP 요청 → 톰캣(서블릿 컨테이너) → 해당 서블릿 실행 → DB 액세스 → HTML 동적 생성 → 응답

서블릿 장단점

장점
  • Java API 모두 사용 가능
  • OS/하드웨어 독립적
  • 스레드 기반 다중 처리
단점
  • HTML 출력 시 out.print() 문자열 기술 → 복잡
  • 디자이너가 화면 수정하기 어려움
  • HTML Form 데이터 처리가 불편
서블릿 생명주기 ★
Ch.4 Section 01
⭐ 핵심: init()은 최초 1회. service()는 매 요청마다.
첫 번째 요청: 컴파일 → init() (초기화) → service() (실행) → doGet/doPost 두 번째 이후: service() → doGet/doPost (init 생략!) 종료/수정: destroy() (정리 작업)
동작 원리: 서블릿 코드를 수정하면 destroy()가 호출되고, 다음 요청 시 다시 init()부터 시작되어 변경사항이 반영됩니다.
메서드호출 시점용도
init()최초 1회DB 연결 등 무거운 초기화 작업
service()매 요청마다GET/POST 판별 및 분기
doGet/Post매 요청마다실제 비즈니스 로직 처리
destroy()종료 시 1회자원 해제 (DB 닫기 등)
service()를 직접 오버라이딩하면 doGet/doPost가 자동 실행되지 않으니 주의하세요!
서블릿 클래스 구조 ★
Ch.4 Section 01
@WebServlet("/LoginServlet") public class LoginServlet extends HttpServlet { protected void doGet(HttpServletRequest request, HttpServletResponse response) { // 1. 응답 설정 (순서 주의: getWriter 전 실행) response.setContentType("text/html; charset=utf-8"); // 2. 파라미터 추출 (리턴 타입: String) String id = request.getParameter("id"); // 3. 출력 PrintWriter out = response.getWriter(); out.print("<h2>안녕, " + id + "!</h2>"); } }
핵심 순서: setContentType() → getWriter() 순서를 지키지 않으면 한글이 깨집니다.
GET vs POST ★
Ch.4 — 설계 원칙과 사용 이유
GET 방식
  • 데이터가 URL에 포함 (조회용)
  • 멱등성: 여러 번 실행해도 결과가 같음
  • 북마크 가능, 길이 제한 있음
POST 방식
  • 데이터가 Body에 숨겨짐 (보안/변경)
  • 서버의 상태를 바꿀 때(글쓰기, 로그인 등) 사용
  • 길이 제한 없음, 대용량 데이터 전송 가능
교수님 팁: "조회는 GET, 생성/수정/삭제는 POST를 쓰는 것이 RESTful한 설계의 기본입니다."

왜 정보 공유가 필요한가 ★
Ch.5 Section 01
HTTP는 비연결형(Stateless) 통신. 페이지를 이동하면 이전 페이지의 정보가 사라짐!

로그인 상태 유지, 장바구니, 쇼핑몰의 주문 정보 등 — 여러 페이지에 걸쳐 정보를 공유해야 하는 경우가 많음.

심화 포인트: 웹 브라우저는 요청을 보내고 응답을 받으면 연결을 바로 끊어버립니다. 이를 무상태(Stateless)라고 하며, 이를 극복하기 위해 쿠키나 세션 같은 기술이 탄생했습니다.

sendRedirect ★
Ch.5 Section 01

서블릿에서 다른 페이지로 이동시키는 메서드. HTTP Response 객체에 있음.

response.sendRedirect("main.jsp"); return; // 반드시 return으로 로직 종료 필요
동작 원리: 서버가 브라우저에게 302 상태 코드를 보내서 "다시 요청해라"고 시키는 방식입니다. 따라서 주소창이 변하고, 이전의 request 객체는 사라집니다.
쿠키 (Cookie) ★
Ch.5 Section 02
⭐ 쿠키 = 서버가 클라이언트(브라우저)에 저장하는 데이터. name=value 형식.
핵심 디테일: 쿠키 값에는 공백, 괄호, 등호, 한글 등을 직접 넣을 수 없습니다. 한글을 저장하려면 반드시 URLEncoder.encode()로 인코딩해야 합니다.

쿠키 생성 & 전송

Cookie ck = new Cookie("name", "kim"); ck.setMaxAge(3600); // 1시간 유지 response.addCookie(ck);
장점
  • 서버 메모리 사용 안 함
  • setMaxAge로 반영구 유지 가능
단점
  • 보안에 취약 (사용자 수정 가능)
  • 4KB 용량 제한
세션 (Session) ★
Ch.5 Section 03
⭐ 세션 = 서버에 저장되는 데이터. 브라우저마다 고유한 세션 ID가 발급됨.
교수님 강조: 세션은 서버에 저장되지만, 누가 누구인지 구별하기 위해 JSESSIONID라는 쿠키를 클라이언트에 보냅니다. 즉, 세션도 내부적으로는 쿠키 기술을 이용합니다.

세션 생성 & 저장

HttpSession session = request.getSession(); session.setAttribute("loginUser", "kim");

세션 읽기 (형변환 필수!)

String user = (String) session.getAttribute("loginUser");
"브라우저를 끄면 세션이 사라지는 이유는 브라우저가 관리하던 JSESSIONID 쿠키가 삭제되기 때문입니다."
RequestDispatcher (Forward) ★
Ch.5 Section 05

request, response 객체를 그대로 다른 서블릿에 전달. 서버 내부에서 이동.

sendRedirect
  • 주소창 바뀜
  • request 공유 안 됨
  • 속도가 약간 느림
Forward
  • 주소창 안 바뀜
  • request 객체 공유
  • 서버 내부 이동이라 빠름
로그인 후 메인으로 보낼 때는 Redirect를, 데이터를 담아 결과창을 보여줄 때는 Forward를 주로 씁니다.
